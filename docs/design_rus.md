# Design Document: JobFinder AI (Курсовая работа)

## 1. Введение (Introduction)

**Задача:** Оптимизация поиска работы для выпускников ВУЗов в Республике Беларусь с использованием нейронных сетей.

**Основная идея:** 
Переход от синтаксического поиска (по ключевым словам) к семантическому (по смыслу). Система анализирует неявные связи между навыками выпускника и требованиями работодателя, используя векторные эмбеддинги и RAG-архитектуру (Retrieval-Augmented Generation) для предоставления интеллектуальных ответов.

**Стек технологий:**
- **Язык:** Python 3.10+
- **Frontend:** Streamlit
- **Backend API:** FastAPI
- **ORM:** SQLAlchemy (асинхронный драйвер `asyncpg`)
- **База данных:** PostgreSQL + pgvector
- **Нейросетевое ядро:** Sentence-Transformers (модель `paraphrase-multilingual-MiniLM-L12-v2`)
- **LLM:** Llama-3-8B (через API Groq)

---

## 2. Пользовательские истории (User Stories)
- **Умный профиль:** Как выпускник, я хочу описать свои навыки, знания и проекты, чтобы система нашла вакансии, где этот опыт будет максимально полезен.
- **Интеллектуальный анализ:** Как пользователь, я хочу знать, почему мне подходит эта вакансия и что конкретно в ней предлагается (условия, требования, стек).
- **Топ-5 и Summary:** Как пользователь, я хочу получать список из 5 наиболее подходящих вакансий с кратким резюме (summary) от LLM по каждой из них, чтобы быстро ознакомиться с ключевыми деталями каждой позиции без необходимости читать полные описания.

---

## 3. Функциональные требования
- **Автоматизированный сбор данных:** Парсинг вакансий с rabota.by с глубокой фильтрацией (Junior/Стажер).
- **Семантическая векторизация:** Преобразование описаний в векторы размерностью 384 (математическое представление смысла текста в 384-мерном пространстве для точного сравнения).
- **Векторный поиск:** Использование косинусного сходства для поиска релевантных вакансий в БД.
- **Валидация данных:** Строгая типизация входящих запросов с помощью Pydantic моделей.

---

## 4. API Design (FastAPI)

Взаимодействие интерфейса и серверной логики осуществляется через REST API:

- **GET `/api/v1/search`** — Принимает запрос пользователя, векторизует его и возвращает список ID вакансий из PostgreSQL.
- **POST `/api/v1/analyze`** — Принимает ID найденных вакансий и профиль пользователя, возвращает структурированный анализ и summary от LLM.
- **POST `/api/v1/ingest`** — Технический эндпоинт для запуска процесса обработки новых вакансий (векторизация).

## 5. Модель сущностей (PostgreSQL)

**Table `vacancies`**:
- `id`: Integer (PK).
- `title`: String (Название вакансии).
- `company`: String (Название компании).
- `salary`: String (Размер ЗП).
- `payment_frequency`: String (Частота выплат).
- `experience`: String (Требуемый опыт).
- `employment`: String (Тип занятости).
- `hiring_format`: String (Формат найма).
- `schedule`: String (График работы).
- `hours`: String (Рабочие часы).
- `work_format`: String (Формат работы: офис/удаленка).
- `skills`: Text (Список ключевых навыков).
- `url`: String (Ссылка на вакансию).
- `description`: Text (Полное описание).
- **`embedding`**: **Vector(384)** — Поле для семантического поиска.

## 6. Техническая реализация (Внутренние процессы)

### Шаг А: Процесс обогащения данных (Ingestion)
1. **Parser:** Скрипт-сборщик наполняет базу данных вакансиями, оставляя поле вектора пустым.
2. **Embedder:** Фоновая задача выбирает записи без векторов, формирует текстовое представление вакансии на основе заголовка, навыков и описания, после чего генерирует вектор через нейросеть Sentence-BERT и сохраняет его в БД.

### Шаг Б: Процесс обработки запроса (Inference)
1. **User Input:** Пользователь вводит текстовое описание своего опыта в интерфейсе Streamlit.
2. **FastAPI Request:** Клиентская часть отправляет этот текст на бэкенд для обработки.
3. **Query Vectorization:** Бэкенд преобразует текст пользователя в числовой вектор той же размерности (384), что и у вакансий в базе.
4. **Similarity Search:** Бэкенд выполняет поиск в PostgreSQL, ранжируя вакансии по близости их векторов к вектору запроса пользователя и выбирая 5 наиболее релевантных.
5. **RAG Logic:** Бэкенд передает в LLM запрос пользователя и данные найденных вакансий для подготовки кратких резюме (summary) и обоснования выбора.
6. **Response:** Итоговый результат с анализом и списком ссылок отображается пользователю в интерфейсе.

---

## 7. Технические нюансы и оптимизация
- **SQLAlchemy + pgvector:** Использование специализированного расширения для PostgreSQL обеспечивает эффективную работу с векторными типами данных и высокую скорость поиска.
- **Prompt Engineering:** Системный промпт LLM настроен на роль эксперта по карьерному консультированию на рынке труда Республики Беларусь.
- **Синхронизация процессов:** Процесс векторизации базы данных отделен от процесса пользовательского поиска, что исключает задержки в работе интерфейса.
- **Интерфейс:** Streamlit обеспечивает реактивное отображение данных и хранение состояния текущего сеанса поиска.
